The Automated Deployment Engine

How I transformed a manual white-label workflow into a governed configuration engine, increasing operational throughput by 400% and helping the team scale from 50+ to 90+ banking partners.

Overview

At Personetics, onboarding a new banking partner required more than 60 hours of manual design work across brand styling, component overrides, documentation, and visual QA. As the partner ecosystem grew from 50+ banks toward 90+ partners, this service-heavy model became a major operational bottleneck.

The objective was to turn repeated manual setup into a productized deployment workflow. I helped define and design a configuration-driven system using a custom Figma widget, a governed Master Vanilla library, and structured variable logic that could apply partner branding without breaking component integrity.

Deployment Engine Overview⚿ NDA-Protected Visual

The widget control plane that turned repeated white-label setup into structured configuration.

The Problem

The team was spending significant time on manual pixel-pushing before clients could review a realistic prototype. Every partner setup required repeated interpretation, manual mapping, and design-to-development clarification. This created long feedback loops, inconsistent component usage, and repeated revision cycles between design, product, engineering, and partner teams.

The larger issue was governance. Client-specific variants were often introduced without being reintegrated into the shared system, and teams were unclear whether Figma, code, or project-specific files were the real source of truth.

“Manual setup had become the product bottleneck.”

Manual Setup Debt⚿ NDA-Protected Visual
Source-of-Truth Gaps⚿ NDA-Protected Visual

Process & Exploration

I partnered with the CPO, design, and engineering leads to audit the full partner onboarding lifecycle. We mapped where time was being lost, which component decisions were repeated, and which variables needed to become controlled configuration inputs.

From that audit, I helped define the Master Vanilla framework: a clean canonical library of 70+ core components prepared for automated theming. Each component was mapped to global variables, semantic logic, and implementation rules so the system could safely apply partner brand changes without component detachment.

The Solution

The final solution replaced manual interpretation with a governed configuration workflow. A custom Figma widget became the control plane for partner setup, allowing brand inputs to be applied through a secure walled-garden interface.

Instead of manually restyling every screen, the widget injected brand variables into the Master Vanilla system, applied changes across 70+ components, and provided faster visual validation. Partners could see meaningful brand results earlier, while internal teams kept control over accessibility, structure, and component logic.

The system helped increase operational throughput by 400%, supported partner growth from 50+ to 90+ banks, reduced support friction, and lowered CSS-related Jira tickets by 80%. Most importantly, it gave the team a scalable model for white-label delivery without increasing design headcount.

The 3-Tier Architecture: Scaling Fintech Deployment

How I refactored fragmented Figma components into a governed 3-tier token architecture, reducing visual deployment from 60 days to 48 hours and creating a shared design-to-code language across Web, iOS, and Android.

Overview

As Personetics expanded across native iOS, Android, and independent desktop platforms, the design system reached a technical ceiling. The workflow had improved, but production delivery still depended on manual translation between Figma variables, component structure, and implementation logic.

This created a translation tax. Even small visual brand updates could take up to 60 days to move from design intent into production. The goal was to create a 3-tier token architecture that could serve as a platform-agnostic source of truth for design, engineering, and future AI-assisted workflows.

TOKEN ARCHITECTURE OVERVIEWⓘ NDA-PROTECTED VISUAL

The Problem

The issue was not only naming inconsistency. Components were split into multiple parallel versions for the same element, which made states, sizes, icons, and interaction rules harder to govern. A button, for example, could exist as separate component groups instead of one unified model, increasing maintenance effort and creating drift between design and code.

Without a unified schema, developers had to manually interpret design specs, mobile and web teams used different variables for the same brand assets, and hard-coded visual values created long-term maintenance debt. The fragmented naming system also made the architecture harder for AI-assisted workflows to understand or maintain.

“Design intent and component logic had drifted apart.”

BEFORE: COMPONENT SPRAWLⓘ NDA-PROTECTED VISUAL

Process & Exploration

I worked with R&D leads to audit component blocks, production assumptions, variable usage, and naming conventions. The audit created a structured inventory of components, variants, statuses, and implementation notes, exposing where Figma definitions and production logic had diverged.

From there, I helped define a 3-tier token hierarchy: primitives for raw values, semantic tokens for intent, and component tokens for scoped behavior. The system was documented through a shared schema and migration framework so existing partners could move into the new architecture without breaking live experiences.

COMPONENT AUDIT & TOKEN MAPPINGⓘ NDA-PROTECTED VISUAL

The Solution

The new model consolidated parallel component groups into unified, token-driven components. States such as idle, hover, focus, loading, disabled, and pressed were defined once and governed through semantic and component tokens, reducing duplication while preserving flexibility for size, icon, and variant needs.

The architecture separated brand identity from core application logic. Partner teams could configure visual expression, while the platform protected accessibility, component behavior, naming structure, and implementation integrity. This also created a stronger foundation for AI-ready workflows because the naming model became consistent, semantic, and machine-readable.

AFTER: UNIFIED COMPONENT MODELⓘ NDA-PROTECTED VISUAL

Outcome & Impact

The 3-tier architecture reduced visual deployment from 60 days to 48 hours, a 97% reduction in lead-time. It also enabled 80% automated brand configuration, improved parity across Web, iOS, and Android, and reduced naming conflicts by moving component behavior into a governed token model.

The work helped move the organization from a manual service model toward a scalable platform model, where visual updates could be governed, validated, and deployed without repeated interpretation between design and engineering.

Intelligence-Driven Product Design & Strategy

How I helped transform Smart Budgeting from a reactive insight experience into a proactive financial utility, driving a 190% engagement increase for a major UK banking partner.

Overview

Personetics' legacy financial tools often behaved like a black box. They surfaced AI-driven insights, but those insights were frequently disconnected from how people actually manage money. This was especially visible in the budgeting experience, which was locked to rigid calendar-month cycles and lacked the flexibility users needed to reflect real income and pay patterns.

The project focused on bridging that trust gap. The goal was to move from reactive notifications toward a permanent, useful financial management home, centered on Smart Budgets and designed to scale across 90+ banking partners and 30M+ end users.

SMART BUDGETING HOMEⓘ NDA-PROTECTED VISUAL

The Problem

The budgeting experience was underperforming against the industry benchmark and user expectations. Research showed that the issue was not only visual clarity, but behavioral logic. Users did not trust automated categories, predictions, or budget states when they did not match their real financial lives.

Budgets were tied to calendar months instead of income cycles, access was driven mainly by passive AI insights, and the system used fragmented logic for different budget states. This made the experience feel reactive, inaccurate, and difficult to rely on as a daily financial tool.

“Trust failed when the product logic did not match real financial behavior.”

Budget Onboardingⓘ NDA-PROTECTED VISUAL
Rigid Setup Stateⓘ NDA-PROTECTED VISUAL
Category Conflict Feedbackⓘ NDA-PROTECTED VISUAL

Process & Exploration

I led research and product framing through JTBD workshops, client performance reviews, and cross-functional analysis. The team focused on understanding where trust broke down and which product behaviors needed to change.

The process identified five core shifts: align budget periods with pay cycles, use clearer semantic language, support AI-assisted setup, give users granular category control, and unify fragmented budget states into one scalable tracker.

The Solution

The redesigned Smart Budgeting model introduced income-aligned budget periods, allowing users to define cycles that match when money actually enters their account. Ambiguous labels like On Track were replaced with clearer factual states such as Within Budget, reducing the gap between AI interpretation and user understanding.

The solution also introduced AI-assisted setup, granular subcategory controls, and a unified budget tracker that could handle different budget configurations within a single scalable use case. This made the product feel less like a one-off insight and more like a permanent financial utility.

The redesign helped shift Smart Budgeting from a reactive insight pattern into a proactive financial management experience. The standardized Smart Budgeting framework also supported global scalability across 90+ banking partners and 30M+ end users, creating a stronger product foundation for future AI-driven financial wellness features.
